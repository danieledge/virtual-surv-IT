RELEASES SHIPPED
----------------
- 0.7.8  Reviews now coach "vibe-coded" code. /deep-review, /audit-review and /remediate ask
         at intake whether the code was AI-assisted; if so, the report adds a "Prompting
         guidance" section - why the top findings happened, plus reusable example prompts for
         a better first draft next time.

- 0.7.9  Memory is project-scoped, not plugin-scoped. This was the important one. Because the
         plugin is installed across many independent projects, it must hold NO project memory -
         otherwise one project's learnings leak into all the others. So project-specific
         learnings (typologies, thresholds, FP drivers) now live in the working project's OWN
         CLAUDE.md; only general, cross-project conventions stay in the plugin's house-rules.
         Re-pointed all 13 agents that recommend lessons, and moved the wash-trade demo
         specifics out to the demo's own scenario-learnings file.

- 0.7.10 Honest self-assessment after an Anthropic-guidance audit (see below).


BEST-PRACTICE AUDIT (vs Anthropic's published guidance + the Claude Code platform docs)
----------------------------------------------------------------------------------------
- Verdict: broadly conformant - genuinely so on the high-value lessons (orchestrator-worker
  delegation, model tiering, token-cost awareness, safety-via-hooks, a real eval harness).

- We corrected our own overstatements rather than gloss them: "advisors are read-only" is now
  "advisors hold no Write/Edit (Bash is execution-gated)"; subagent self-assessment and
  condensed sub-agent returns are marked honest partials; and the roster is reframed as a
  LIBRARY, not a pipeline - the PM engages the minimal sufficient subset per task.

- We checked the platform auditor's two "broken packaging" alarms and found them NOT
  substantiated (the hooks auto-load, the manifest paths resolve, the plugin demonstrably
  installs) - so no disruptive restructure on a false alarm.

- On "should we benchmark the team size?" - we concluded it's the wrong question. You never
  engage all 16; the PM selects what a task needs. "Simplest thing that works" is satisfied by
  that selection, not by a benchmark, so we documented it honestly instead of building one.
