01 Why this exists

The problem
Surveillance runs on unglamorous work: BRDs and technical specs, traceability matrices, threshold-tuning evidence, model-validation packs, audit responses, reconciliations - each needing a rare mix of fin-crime knowledge, software engineering and audit-grade writing. So backlogs of scenario changes and documentation debt grow faster than teams can shrink them, while regulator expectations keep rising.
Why AI fits this problem
Nearly everything the field produces is a language artifact - requirements, findings, traceability, validation reports. Large language models are strongest at reading code and documents at volume and writing structured, evidenced prose against a stated standard. Work that costs salaries and lead time in the human world costs tokens and minutes here.
Why it's credible here
Surveillance is a controls culture - segregation of duties, independent review, gates. Raw AI has none of that discipline, so this setup imposes it mechanically: the field's own governance habits, translated into architecture.
Do we need prompt-engineering experts? No
The prompting craft is built in, not user-supplied. Every prompt that matters here - agent definitions, workflows, templates, critique standards - is a versioned, tested artifact: an improvement is committed once and every future engagement benefits. Users answer the project manager's structured questions in plain English. The wider industry is moving the same way: modern models are far less sensitive to clever phrasing, and the discipline has shifted from wording tricks to context engineering (controlling what information the model sees), which this setup handles in its process. The expertise a compliance engineering team needs is the one it already has: knowing what good looks like in its own field.

03 What it produces

It builds as well as audits. The same team that assesses an estate can define the requirements, write the technical specification, implement the detection logic, test it and document the result - the full lifecycle, with every deliverable traced to the regulatory obligation that motivates it.

Requirements & specification
Elicitation, BRDs and functional specs to industry method (BABOK, EARS, Gherkin) - every requirement traceable through design, code and test to the obligation.
Detection scenarios
Transaction-monitoring and market-abuse detection logic - specified, SME-challenged, built, and tested with true-and-false-positive cases.
Tuning & validation
Threshold calibration (ATL/BTL), segmentation, and independent model validation - the volume-versus-coverage trade-off quantified, not guessed.
Code & audit reviews
Deep code review, performance review, security audit and requirements-vs-code assessments - findings scored, dispositioned and evidenced.
Data engineering
Pipelines, ETL, reconciliation and reporting jobs - plus a data-preparation front door that masks or synthesises data before any model sees it.
Audit-ready documentation
Document control, scope at a stated commit, dispositions and a limitations statement on every review - consumable directly by second line and audit.

04 Cost - the mechanics that manage it

Cost control is engineered into the process, not asserted. Seven mechanisms, each inspectable:

Not everything is an LLM
A surprising amount of the work is ordinary, deterministic software: static analysers and security scanners, the data-masking pipeline, document rendering, citation and completeness gates, calibration scripts. These run at effectively zero model cost and never hallucinate - their output is what the 📊 measured tag means. The model is reserved for what only it can do: reading, synthesis and judgement.
Dormant by default
Nothing loads until a workflow is invoked - an installed team consumes no tokens between engagements. No seats or licences beyond the Claude subscription the team runs on.
Team size stated before spend
The project manager names which specialists a job needs - typically 2-5 of the 17 - before fanning out, and the decision can be vetoed. "Simplest thing that works" is a tested behaviour, sampled by the evaluation harness.
The right model for the job
Not every task needs the most capable (and most expensive) model. Each specialist is assigned a model tier to match its work: mechanical steps like scoring, tallying and filtering run on a small, fast model at a fraction of the unit cost, while the frontier tier is reserved for the judgement that earns it. The assignment is per-agent configuration, visible and changeable.
Context is bounded, not cumulative
A long engagement never becomes one ever-growing conversation. Each specialist starts fresh with a scoped brief and only the documents it needs, then returns a short distilled summary rather than its whole working transcript. The full detail lives in the shared artifacts, so context resets between agents instead of compounding - the single biggest cost and reliability lever in multi-agent work.
Footprint on every engagement
The closing email states the engagement's approximate token spend and agent count - a required line in the template, so the multi-agent multiplier is tracked, never hidden.
Warm starts
A per-project codebase map means repeat engagements build on recorded knowledge instead of re-exploring the estate - external research suggests memory pays back after roughly ten uses.
Worked example - a real engagement footprint
A full transaction-monitoring implementation review, traced line-by-line against its technical specification document and delivered as an audit-quality findings report, cost roughly $7.50 of model usage. The equivalent human effort - days of a contractor who combines fin-crime domain knowledge with code fluency - prices in thousands. The gap is the value the existing team keeps: engineers spend their time on judgement and sign-off, not document drafting.

05 Why it can be trusted

One question deserves one chapter: why trust output like this? The answer is split into what is enforced by hard technical controls and what is enforced by process - and the two are never conflated.

Hard controls - enforced mechanically

Raw data is unreachable
Client raw data is structurally blocked from the model; work happens on masked or synthetic data via a keyed pipeline.
HARD - hooks + OS permissions + CI
How: a PreToolUse hook rejects reads under data/raw/; OS-level permission denies back it for the file tools; a CI job fails any commit containing raw data.
Humans hold the keys
The team cannot run code, grant itself permissions, or edit its own guardrails - consent is a file only a human can create; nothing touches a live system without sign-off.
HARD - consent gate, model-write blocked
How: execution is gated on a consent file; a second hook blocks the model's own tools from ever creating that file or editing the guards and settings.
Fail closed
If a safety control crashes, it blocks rather than silently allowing. The guards are regression-tested against malformed input.
HARD - crash wrappers + tests
How: guard crash wrappers exit with the blocking code on any exception, so a crashed control denies by default; malformed-input regression tests pin this.
Segregation of duties
Reviewers, SMEs and the model validator physically cannot edit code - read-only tool grants, not convention. A checker never marks its own homework.
HARD - tool grants
How: reviewer agent definitions omit Write/Edit from their tool grants; the platform enforces grants per agent, so a reviewer cannot modify files at all.
Evidence, not claims
Every finding is tagged measured 📊 or inferred 🧠; citations come from a verified register; limitations are stated on every review.
PROCESS - gated + eval-sampled
How: citations resolve through a checker script against a verified register; findings carry file:line plus measured/inferred tags, checked at the delivery gate.
Self-tested, and self-checked as it evolves
The team is itself software under continuous integration: every change to it must pass 360+ tests on multiple platforms, lint and security analysis, a secret scan, a no-raw-data check and plugin-manifest validation before it lands - plus the golden-case eval harness that catches output-quality drift. Its residual risks are documented in a public threat-model decision record.
HARD - CI on every change
How: GitHub Actions runs the test matrix, ruff/bandit/shellcheck, a gitleaks secret scan and a no-raw-data check on every push; a red run blocks the change.

Process controls - gated, evidenced, eval-sampled

The hallucination question, answered up front
Language models can state things confidently that are not true, and no honest assessment claims otherwise. This setup treats that as a design constraint and controls it in two parts. First, findings are always evidenced. Every claim cites the actual code it rests on - file and line, config key, commit - and ships as a reviewable artifact, so a reviewer checks the source rather than trusting the prose. Everything is labelled 📊 measured or 🧠 inferred, and presenting an inference as fact is an automatic failure in the evaluation harness. Second, regulatory citations come only from a verified register, never from model memory. Transparency matters here: that register is deliberately small - a set of checked sources, not a comprehensive regulatory database. If an obligation is not in it, the team says so rather than inventing one. Access to authoritative sources such as the FCA Handbook is being explored to grow it. Independent review and human sign-off remain in the loop; residual risk does not reach zero - the controls make errors findable and attributable instead of fluent and invisible.
How: check_citations flags any citation not in the register; the eval rubric auto-fails inference presented as fact; a mechanical gate rejects findings without an impact line.
Critiqued against named standards
Before handover, a specialist who is not the author checks each deliverable against its profession's criteria: audit findings → the 5 C's (criteria · condition · cause · consequence · corrective action) · requirements → BABOK quality criteria · QA evidence → ISO/IEC 29119-shaped completeness. Generic "look it over again" passes are banned - research shows critique only helps with an external standard.
How: each template states its checking standard in its header; the operating guide requires the critic is never the author; the delivery gate records which standard was applied.
Anchored on gold exemplars - a library that is still growing
Writers anchor on known-good specimen documents rather than rules alone; the test of a good finding is that a reader who was not in the session understands what's wrong, why, why it matters, and what to do. To be clear about maturity: the exemplar and template library is young and deliberately evolving - each engagement's lessons are folded back into the templates (a live engagement did exactly that this week), so the standard improves with use. It is a growing practice, not a finished product.
How: known-good specimen documents live in the repository and the writing workflows are instructed to anchor on them before drafting.
Regression-guarded
Document quality is eval-tested like code: golden cases fail a review that names a defect without its cause and impact, even when detection succeeded - so a quality lapse is caught by the harness, not by the client.
How: golden eval cases score sampled outputs against per-deliverable rubrics; a review whose findings lack cause and impact fails the case even when detection succeeded.
Definition of Done
No delivery closes without traceability, independent QA, review dispositions, documentation and recorded human approval - checked at a gate, partly by machine.
How: a gated checklist the project manager attests, plus a mechanical script verifying rendered artifacts, the closing email, finding impact lines and memory hygiene.
Consumable by audit
Outputs are structured for model-governance and audit review - while making no regulatory-compliance claims. Framework mapping stays the compliance function's call.
How: the review template's fixed skeleton (document control, commit-pinned scope, dispositions, limitations) is the default output shape at every review depth.
Transparent by design
Iteration history, filtered-findings counts, tooling coverage and token footprint are all on the record - what the team did not do is stated as clearly as what it did.
How: append-only iteration logs and test-cycle tables are built into the templates; filtered-finding counts, tooling coverage and the cost footprint are mandatory sections.

06 Where this stands

This is a working proof of concept. It assists the engineering behind surveillance; it is not compliance, legal or regulatory advice, and it makes no framework-compliance claims. Detection content is illustrative until validated on real data, and nothing deploys to a live system without the owning team's approval. Where a safeguard is enforced by process rather than a hard control, this brief says so - hopes are not controls.

The fastest way to evaluate it is not this document: hand it one real item from the backlog - a scenario change, a tuning review, an assessment against a specification - and judge the delivered artifacts against the specimens above. The engagement will report its own cost footprint in its closing email.

07 Glossary

ATL/BTL - above-the-line / below-the-line testing: sampling either side of a detection threshold to evidence it is set neither too loose nor too tight.
BABOK - Business Analysis Body of Knowledge; the professional standard for requirements work.
BRD - Business Requirements Document: what the business needs, and why.
EARS - Easy Approach to Requirements Syntax; sentence patterns that force requirements to be testable.
5 C's - the audit profession's structure for a finding: criteria, condition, cause, consequence, corrective action.
Gherkin - Given/When/Then phrasing for acceptance tests.
ISO/IEC 29119 - the international standard for software-testing documentation.
LLM - large language model; the AI technology the team runs on.
OWASP - Open Worldwide Application Security Project; the reference standards for security review.
PM - project manager - here, "Morgan", the orchestrating agent that runs each engagement.
QA - quality assurance: independent testing with evidence.
RTM - Requirements Traceability Matrix: requirement → design → code → test → regulatory obligation.
SAR - Suspicious Activity Report, filed with the financial-crime authorities.
SME - subject-matter expert.
TM - transaction monitoring: detection of suspicious activity in transaction flows.
TSD - Technical Specification Document: how the requirement is implemented.
